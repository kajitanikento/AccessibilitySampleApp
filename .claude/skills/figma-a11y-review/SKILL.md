# Figma アクセシビリティレビュースキル

## 概要

Figmaデザインをチェックリストに基づいてレビューし、Markdownレポートを自動生成する。

---

## 実行手順

### Step 1: 入力の解析

ユーザーのメッセージから以下を抽出する：

- **Figma URL**: `https://www.figma.com/design/:fileKey/...?node-id=:nodeId` の形式
  - `fileKey`: URLの `/design/` 直後のセグメント
  - `nodeId`: `node-id=` パラメータの値（`1-2` → `1:2` に変換）
- **画面名**: URLやユーザーのメッセージから推測。不明な場合はFigma取得後にデザインから判断する。

### Step 2: チェックリストの読み込み

`.claude/skills/figma-a11y-review/references/ui-design-accessibility-check-list.md` を Read ツールで読み込む。

CSVの列構成: `ID, チェック内容, 重篤度`

重篤度の定義:
- `[CRITICAL]`: 重大な問題。リリース前に必ず修正が必要
- `[MAJOR]`: 重要な問題。可能な限り修正が必要
- `[NORMAL]`: 通常の問題。対応を推奨
- `[MINOR]`: 軽微な問題。余裕があれば対応

### Step 3: Figmaデザインの取得

`mcp__figma__get_design_context` ツールを呼び出す：
- `fileKey`: Step 1 で抽出した値
- `nodeId`: Step 1 で変換した値
- `clientLanguages`: `swift`
- `clientFrameworks`: `swiftui`

取得したスクリーンショットとコードを元にデザインを把握する。

### Step 4: アクセシビリティレビュー

チェックリストの各IDについて、取得したデザインを分析する。

#### 分析のポイント

**コントラスト比（ID: 0001, 0003）**
- コード中の色値（`text-[#xxxxxx]`, `bg-[#xxxxxx]`, `text-[color:var(--...,#xxxxxx)]`）を抽出
- 文字色と背景色のペアを特定し、コントラスト比を推定計算する
- sRGB相対輝度: `L = 0.2126 * R_lin + 0.7152 * G_lin + 0.0722 * B_lin`
  - 各チャンネル: `c_lin = ((c/255 + 0.055)/1.055)^2.4`（c/255 > 0.04045の場合）
- コントラスト比: `(L_lighter + 0.05) / (L_darker + 0.05)`
- 判定基準: 12px通常テキスト → 4.5:1以上 / 18pt以上または14pt以上Bold → 3:1以上

**タップターゲット（ID: 0333, 0334）**
- ボタン・フォームコントロールの高さ（`h-[Xpx]`）を確認
- iOS最小タップターゲット: 44×44px
- アノテーション（`data-accessibility-annotations`）でタップ領域の仕様が明示されているか確認

**アクセシビリティ仕様（ID: 0581）**
- `data-accessibility-annotations` 属性の有無と内容を確認
- 読み上げラベル・ロール・ステート・アナウンス文言の仕様が記載されているか確認
- `[コンポーネント共通]` 注記でコンポーネント全体への適用が明示されているか確認

**ステータスメッセージ（ID: 1171）**
- 動的変化（完了・削除・エラー等）発生時のアナウンス文言が設計資料に明示されているか確認

**アイコン・画像（ID: 0391, 0421, 0451）**
- アイコンに `alt=""` か意味のある `alt` テキストが設定されているか確認
- テキストラベルのないアイコンボタンにアクセシビリティラベル仕様があるか確認

**見出し（ID: 0541, 0722）**
- 見出し要素（`h1`, `h2` 等のコンポーネント名またはアノテーション）の有無を確認

**色だけに依存しない表現（ID: 0031, 0032）**
- 状態・情報をテキストとアイコンの組み合わせで表現しているか確認

**ボタン・リンクのラベル（ID: 0241, 0242）**
- 同一画面内で同じラベルのボタンが複数ある場合、文脈付きラベル仕様があるか確認

#### 判定基準

各チェック項目を以下のいずれかに分類：
- `✅ 問題なし`: チェック内容を満たしている
- `⚠️ 要確認`: デザインから判断できない（実装時に確認が必要）
- `❌ 問題あり`: チェック内容を満たしていない（修正が必要）
- `— 対象外`: 当該画面には該当する要素がない、またはスコープ外

### Step 5: レポートの生成

#### 出力先ファイルパス

```
report/figma-a11y-review/{画面名}/{レビューした日付}.md
```

- **画面名**: Figmaのデザインから判断したスネークケースの画面名（例: `task-list`, `task-detail`, `wcag-demo`）
- **日付**: レビュー実施日時を日本時間で `YYYY-MM-DD-HH-mm` 形式で使用（例: `2026-03-10-14-30.md`）
- ディレクトリが存在しない場合は自動作成する

#### レポートフォーマット


## 注意事項

- チェックリストの全IDについて漏れなくレビューすること
- コントラスト比は推定計算でよいが、計算根拠（色値）を明示すること
- `data-accessibility-annotations` の内容は仕様として正として扱う
- デザインに存在しない要素（音声・映像等）は「対象外」として記録する
- 複数の画面（nodeId）がある場合は画面ごとに個別レポートを作成する
- タイムスタンプを含むためファイル名が衝突することはない
